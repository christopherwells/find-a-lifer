1. Overview & Design Philosophy

Find-A-Lifer is a progressive web app that helps birders locate species they haven't yet observed ("life birds"). It combines eBird community science data with an interactive map to show where and when species occur, at weekly temporal resolution across a multi-scale hexagonal grid.

All species occurrence data is pre-computed offline and served as static JSON files — there is no backend at runtime. This architectural choice enables offline support via service workers, instant loading from CDN/GitHub Pages, and zero server costs. The trade-off is that the data pipeline must produce complete, self-contained output files.

Why static pre-computation? The app is designed for field use by birders who may have limited connectivity. Pre-computing all data means the app works offline after initial load. It also means anyone can host a copy on GitHub Pages with no infrastructure. The pipeline processes ~50GB of raw eBird data into ~500MB of static JSON.

2. Input Data & Filtering

2.1 Data Source

All occurrence data comes from the eBird Basic Dataset (EBD), a global community science database with over 1.5 billion bird observations. Regional subsets are downloaded as paired files:

  • Sampling events file — metadata for each checklist (location, date, effort, protocol)
  • Observations file — species detections linked to sampling events

Processing is two-pass: first extract valid checklists from the sampling file, then match observations against those checklists. This avoids loading the much larger observations file into memory.

2.2 Checklist Effort Filters

Not all eBird checklists are suitable for computing reporting frequencies. We apply strict effort filters to retain only standardized, effortful surveys:

FilterCriterionValue
Date rangeObservation year1900–2025 (effectively unlimited)
ProtocolSurvey typeStationary or Traveling only
CompletenessAll species reportedYes (complete checklists only)
Duration (min)Survey length≥ 30 minutes
Duration (max)Survey length≤ 300 minutes
Distance (max)Traveling distance≤ 8 km
Why these filters?
  • Complete checklists only: Reporting frequency (detections ÷ checklists) is only meaningful when observers report all species seen, not just highlights. Incomplete lists inflate frequency for rare/exciting species and deflate it for common ones.
  • Stationary + Traveling only: These are eBird's standardized protocols with consistent effort. "Incidental" observations (seeing a bird in a parking lot) are inherently biased — the observer wasn't systematically surveying. Even with the 30-minute duration filter, we explicitly exclude incidentals to be safe.
  • 30-minute minimum: Very short checklists undercount species. A 5-minute stop detects only the most conspicuous species, artificially deflating frequencies for quieter or less visible species.
  • 300-minute maximum: Extended surveys (hawk watches, Big Sits) are valuable data, but beyond 5 hours the effort becomes qualitatively different — observer fatigue, changing conditions, etc.
  • 8 km max distance: Checklists covering long distances span too much area to be localized to a single grid cell. At our finest resolution (res 4, ~45 km hex diameter), 8 km is less than 20% of the cell width — still well-localized.
  • Stationary checklists included (distance = 0): A birder standing at a marsh or hawk watch for an hour produces excellent, perfectly-localized data. Earlier versions excluded these via a minimum distance filter, which discarded a large fraction of high-quality observations.
  • No year floor: All years are included. The Historical protocol (which would introduce life-list checklists) is already excluded by the Stationary/Traveling protocol filter. Pre-2006 data is sparse but valid.

Typically 10–30% of raw checklists pass all filters. The primary rejection reasons are incomplete lists, unsupported protocols, and duration outside bounds.

3. Spatial Grid System

3.1 H3 Hexagonal Grid

Species occurrences are aggregated onto Uber's H3 hexagonal grid at three resolution levels:

ResolutionHex DiameterMap Zoom RangePurpose
Res 2~450 km0 – 3.5Continental overview
Res 3~120 km3.5 – 5.5Regional patterns
Res 4~45 km5.5 – 22Local trip planning
Why hexagons? Hexagonal grids have uniform adjacency — every neighbor is equidistant from the center, unlike square grids where diagonal neighbors are √2× farther. This matters for spatial smoothing: when we interpolate frequencies from neighboring cells, each neighbor contributes equally in distance. H3 also provides a built-in hierarchy where each coarse cell contains exactly 7 finer cells, enabling multi-resolution fallback.
Why these resolutions? Resolution 5 (~15 km hexes) was tested but produced too many cells with sparse data, even after smoothing. Resolution 4 strikes a balance — large enough to accumulate meaningful checklist counts, small enough to distinguish local habitats. Resolution 2 provides the "zoomed out" continental view where individual cells don't need to be precise.

3.2 Ocean Handling

Unlike earlier versions that aggressively removed ocean cells, the current pipeline preserves ocean cells with direct observations — pelagic surveys and boat-based checklists produce valid bird data. Instead, ocean integrity is maintained through a multi-layer defense:

  1. Gaussian covariate similarity (§5.3): ocean↔forest similarity is ~0.006, naturally preventing land birds from bleeding into ocean cells through the ensemble.
  2. Well-sampled cell bypass (§5.2a): ocean cells with ≥10 checklists skip the ensemble entirely, using only their direct pelagic survey data.
  3. Week-limited ensemble (§5.2b): ensemble cannot fabricate data for weeks with no direct observations in a cell.
  4. Phase 2 ocean smoothing filter: neighbor smoothing does not create new data in cells with water covariate ≥ 0.9.
  5. Land-obligate species filter (§7.2): species classified as land birds are removed from ocean cells in the final output.

Environmental covariates are computed from EarthEnv 1km land cover rasters. Each H3 cell's water fraction determines whether it is treated as ocean (water ≥ 0.9). Note: cells whose centers fall in open water may have water=1.0 even if their edges clip land — a known limitation of center-point covariate extraction.

4. Frequency Estimation

4.1 Reporting Frequency

The core metric is reporting frequency: the proportion of valid checklists in a cell-week that detected a given species.

frequency(species, cell, week) = detections(species, cell, week) ÷ checklists(cell, week)

This is the same metric used by eBird Status & Trends. It represents the probability that a standardized birding trip in that location and time period would detect the species. A frequency of 0.45 means 45% of qualifying checklists reported the species.

Why reporting frequency instead of abundance? The EBD contains count data (number of individuals), but abundance estimates require sophisticated detection modeling to account for imperfect detection. Reporting frequency is simpler, more robust, and directly interpretable as "chance of seeing this species on a birding trip." It naturally normalizes for observer effort — a cell with 1,000 checklists and one with 50 produce comparable frequencies.

4.2 Temporal Pooling

Many cell-week combinations have too few checklists for a reliable direct estimate. We stabilize frequencies by pooling with adjacent weeks (±1 week, wrapping at year boundaries):

For a species detected in cell C during week W:
  If checklists(C, W) ≥ 5:
    freq = detections(C, W) / checklists(C, W)        # Direct estimate
  Else:
    pooled_det = detections(C, W-1) + detections(C, W) + detections(C, W+1)
    pooled_cl  = checklists(C, W-1) + checklists(C, W) + checklists(C, W+1)
    If pooled_cl ≥ 3:
      freq = pooled_det / pooled_cl                    # Pooled estimate
    Else:
      No estimate produced (insufficient data)
Why pool adjacent weeks? Species occurrence changes gradually — a bird present in week 20 is almost certainly present in weeks 19 and 21. By borrowing checklist effort from neighboring weeks, we can produce estimates for cells that would otherwise have no data for that specific week. The 3-week window is narrow enough to preserve seasonal patterns while broad enough to fill gaps. The minimum threshold drops from 5 (direct) to 3 (pooled) because pooling itself provides some noise reduction.

4.3 Minimum Checklist Thresholds

A cell-week must have at least 5 checklists for a direct frequency estimate, or 3 checklists after temporal pooling. These thresholds prevent unreliable estimates from cells with one or two observations.

Why 5 and 3? With fewer than 5 checklists, a single observation can swing frequency from 0% to 20%+ — too noisy to be useful. The pooled threshold of 3 is lower because temporal pooling already reduces variance by combining multiple weeks. eBird S&T uses similar minimum sample sizes.

5. Stixel Ensemble Smoothing

5.1 Motivation

Even after temporal pooling, frequency estimates from individual cells can be noisy — especially in areas with moderate but not abundant eBird coverage. The stixel ensemble approach, inspired by eBird Status & Trends methodology (Fink et al. 2020), stabilizes these estimates through spatial block-averaging across multiple random partitions.

The ensemble is only applied at resolution 4. Resolutions 2 and 3 have large enough cells (~450 km and ~120 km diameter) that they naturally accumulate sufficient checklist data — averaging ~21,000 checklists per cell at res 3. Applying the ensemble at these scales would over-smooth regional patterns without meaningful noise reduction.

Why an ensemble instead of simple spatial averaging? A simple average of neighboring cells treats all neighbors equally and always smooths toward the mean. The ensemble approach creates different random spatial groupings on each run — sometimes a cell is grouped with its northern neighbors, sometimes its southern ones. Averaging across these random partitions reduces noise without systematically blurring boundaries between habitat types. Covariate weighting further ensures that ecologically similar cells contribute more to each other's estimates.

5.2 Algorithm

The ensemble runs 25 independent passes:

  1. Random partition: Shuffle all cells with data. Greedily form spatial blocks by selecting an unassigned seed cell and adding its H3 k-ring neighbors (k varies by resolution).
  2. Block-level frequency: Pool all checklists and detections within each block. Compute frequency per species per week at the block level.
  3. Weighted assignment: Assign the block frequency back to each cell in the block, weighted by environmental covariate similarity between the cell and the detection cells.
  4. Accumulate: Track the sum and count of frequency assignments per cell across all 25 runs.

After all runs, the ensemble frequency is the average across runs. This is then merged with the direct temporal-pooling estimate:

merged = 0.6 × direct + 0.4 × ensemble

Critically, if a species is present in the ensemble but not detected directly in the cell, the direct frequency is treated as 0.0 (not missing). This means ensemble-only species enter the merged estimate at only 40% of their ensemble frequency, preventing phantom species from appearing at full confidence.

5.2a Well-Sampled Cell Bypass

Cells with ≥10 total checklists (across all weeks) skip the ensemble entirely and use direct frequencies only. These cells have enough local data that ensemble smoothing would only dilute their signal. This is particularly important for preventing ocean cells with pelagic survey data from being contaminated by neighboring land cells' ensemble estimates.

5.2b Week-Limited Ensemble

The ensemble is only merged for weeks where the cell has direct observations. If a cell has checklists in summer but none in spring, the ensemble cannot fabricate spring data for that cell. This prevents the common artifact of cells showing 52-week species presence when they only have seasonal coverage.

Why 60/40 weighting? Direct observations in a cell are the most reliable signal — a bird was actually seen there. But the ensemble estimate is more stable (25 samples vs. one). The 60/40 split trusts local observations while still benefiting from the noise reduction of the ensemble.

5.3 Covariate Weighting

When assigning block-level frequencies to individual cells, each cell's contribution is weighted by how environmentally similar it is to the cells where the species was actually detected. Similarity uses a Gaussian kernel with steepness parameter k=4:

similarity = exp(−d² × 4)
where d² = Σ(cova[k] − covb[k])² over land cover types + (Δelev / 1000)²

Covariates include land cover fractions from EarthEnv (trees, shrub, herb, cultivated, urban, water, flooded vegetation) and elevation. The Gaussian kernel provides sharp discrimination between dissimilar habitats:

ComparisonSimilarityEffect
Same habitat~0.99Free sharing
Ocean ↔ coastal~0.35Shorebirds can cross
Coastal ↔ forest~0.14Moderate
Ocean ↔ forest~0.006Essentially zero
Forest ↔ plains~0.21Some generalist sharing
Why Gaussian instead of inverse distance? An earlier formula (1/(1+d×3)) compressed all similarity scores into the 0.2–0.4 range, barely distinguishing habitats. Ocean↔forest scored 0.228 — far too high, allowing land birds to bleed into ocean cells through the ensemble. The Gaussian kernel drops that to 0.006, effectively blocking cross-habitat contamination while still allowing similar habitats to share freely. If covariates are unavailable, similarity defaults to 0.5 (neutral).

5.4 Block Size

ResolutionEnsembleBlock k-ringApproximate Block Size

Res 425 runsk = 1~7 cells (~14,000 km²)
Why k=1? Larger blocks (k=2, k=3) were tested but produced over-smoothed results — rare or localized species disappeared because their signal was diluted across too many cells. A 1-ring block (~7 cells) provides enough pooling to stabilize estimates while preserving meaningful spatial variation.

6. Spatial Interpolation & Fallback

6.1 Neighbor Smoothing (Res 4 Only)

After the ensemble, resolution 4 applies one additional round of neighbor smoothing to fill cells that still have no data. For each empty cell surrounded by data cells:

For each neighbor N in 1-ring around empty cell C:
  If N has data:
    weight = spatial_weight(distance=1) × covariate_similarity(C, N)

weighted_freq(C) = Σ(freq(N) × weight(N)) / Σ(weight(N))

Smoothed cells are marked as "smoothed": 1 in the output grid and displayed at reduced opacity in the app.

Why only at res 4? Resolutions 2 and 3 have large enough cells that nearly all land area is covered by direct data. Resolution 4's smaller cells (~45 km) leave gaps, especially in less-birded areas. Neighbor smoothing fills these gaps with a conservative, distance-weighted estimate. We don't smooth at coarser resolutions because it would create data in places too far from any observations.

6.2 Multi-Resolution Independence

Each resolution stands on its own data — there is no downward fallback between resolutions. Resolution 2 is aggregated upward from resolution 3 archive data, but resolution 3 does not fall back to resolution 2, and resolution 4 does not fall back to resolution 3.

Why no fallback? Earlier versions used parent-resolution data (with a 30% discount) to fill empty child cells. This caused land bird species to propagate into ocean cells — a res 3 coastal cell containing both land and ocean would push its full species list down into all seven res 4 children, including ocean-only cells. With the ensemble already filling gaps at res 4, fallback was both unnecessary and harmful.

7. Thresholds & Quality Control

7.1 Minimum Smoothed Frequency

Species-cell entries from interpolated (non-direct) cells with frequency below 0.5% are dropped:

If cell is interpolated AND freq < 0.005: discard

This threshold is not applied to direct-detection cells — if a species was actually observed in a cell, it stays regardless of computed frequency.

Why exempt direct detections? A rare vagrant observed once in 200 checklists has a 0.5% frequency — low, but real. Dropping it would erase genuine records. For interpolated cells, though, a 0.3% frequency likely represents noise propagated through the ensemble and smoothing. The threshold strikes a balance between coverage and credibility.

7.2 Land-Obligate Species Filter

To prevent land birds (e.g., Ravens, woodpeckers) from appearing in ocean cells through ensemble bleeding or sparse pelagic checklist noise, a land-obligate species filter is applied in the output phase:

  1. For each species, compute the average water covariate across all cells where it occurs.
  2. Species belonging to waterbird families (gulls, ducks, loons, alcids, shorebirds, cormorants, pelicans, storm-petrels, shearwaters, boobies, frigatebirds, tropicbirds, skuas, oystercatchers, grebes) are exempt — always allowed in ocean cells.
  3. Non-exempt species with average water < 0.3 are classified as land-obligate.
  4. Land-obligate species are removed from cells with water ≥ 0.9.
Why family exemption? Many waterbirds (Herring Gull, Common Loon, scoters) are commonly reported from harbors and lakeshores — cells with low water covariates. Their average water score drops below 0.3 despite being perfectly legitimate ocean species. The family-level exemption ensures all waterbird families pass through regardless of their covariate statistics. For non-waterbird species, the avg_water < 0.3 threshold catches true land birds like passerines, raptors, and owls.

8. Output Format

8.1 Frequency Encoding

Frequencies (0.0–1.0) are encoded as unsigned 8-bit integers (0–255) for compact storage:

freq_uint8 = max(1, min(255, round(freq × 255)))

Client-side decoding: probability = freq_uint8 / 255. A value of 127 represents ~50% frequency.

Why uint8? The raw frequency data for all species, cells, and weeks would be hundreds of megabytes as floating-point JSON. Quantizing to 0–255 roughly halves file sizes while preserving more than enough precision for visualization and trip planning. The ~0.4% granularity (1/255) is well below the noise floor of the underlying estimates.

8.2 File Structure

All output is organized per resolution in data/r{2,3,4}/:

FileContentsPurpose
grid.geojsonH3 hex polygons with cell_id, center coords, label, smoothed flagMap overlay geometry
weeks/week_XX_summary.json[cell_id, species_count, max_freq, n_checklists] per cellFast heatmap rendering
weeks/week_XX_cells.json[cell_id, [species_ids...], [freq_uint8...]] per cellCell click → species list
species-weeks/{code}.json{week: [[cell_id, freq], ...]}Single-species range maps

A shared species.json contains metadata for all species (common name, scientific name, family, taxonomic order, eBird species code).

Why separate summary and cell files? The summary file (~2 MB/week) contains just enough to render the heatmap — species count and max frequency per cell. The full cell file (~30-50 MB/week) contains the complete species list with individual frequencies, but is only loaded on demand when the user clicks a cell. This lazy-loading pattern keeps the initial page load fast while still providing full detail when needed.

9. Client-Side Computation

9.1 Lifer Density

The default view mode subtracts the user's life list from each cell's species count:

lifer_count(cell, week) = total_species(cell, week) − seen_species(cell, week)

9.2 Combined Lifer Probability

For the "Frequency" view, the app computes the probability of observing at least one unseen species during a visit:

P(≥1 lifer) = 1 − ∏i(1 − freqi)   for all unseen species i in the cell
Why this formula? Under the assumption that detecting each species is approximately independent (reasonable for well-separated taxa), the probability of detecting none of them is the product of the individual non-detection probabilities. One minus that product gives the probability of at least one detection. This is the most useful metric for a lifer-seeking birder: "if I go here, what's my chance of adding something new to my list?"

9.3 Resolution Switching

The app automatically switches between resolutions as the user zooms:

  • Zoom 0–3.5: Resolution 2 (~450 km hexes)
  • Zoom 3.5–5.5: Resolution 3 (~120 km hexes)
  • Zoom 5.5+: Resolution 4 (~45 km hexes)

Each resolution has independently computed data, so switching is seamless — the appropriate grid and weekly data are loaded fresh for the new resolution.

10. Grid Cell Labeling

Each grid cell is labeled with the nearest recognizable city for user orientation. The labeling algorithm balances proximity, city size, and label diversity:

10.1 Scoring

score = population / (distance_km + 1)1.5 × region_boost

Where region_boost = 1.5 if the city and cell are in the same state/province, else 1.0. The search radius is resolution-dependent (120 km for res 3, 60 km for res 4).

10.2 Deduplication

To prevent a single major city (e.g., New York City, Boston) from labeling dozens of surrounding cells, each city is limited to a maximum number of cells per resolution (2 at res 3, 3 at res 4). City name variants ("South Boston," "East Boston," "North Boston") are grouped under the base name for deduplication.

Why population-weighted scoring? Users need recognizable landmarks for orientation. "Near Boston" is more useful than "near Framingham" even if Framingham is slightly closer. The 1.5 exponent on distance ensures proximity still matters — a distant megacity won't dominate a cell right next to a midsized town. The same-state boost prevents border cells from being labeled with out-of-state cities (e.g., a New Hampshire cell getting a "Boston, MA" label).

11. Habitat Classification

Each species is assigned habitat labels based on its occurrence patterns across environmental covariates. The pipeline (compute_species_habitat.py) computes frequency-weighted average covariates for each species across all cells and weeks where it occurs.

11.1 Covariate Sources

Cell-level environmental covariates are extracted from two sources:

  • Land cover: EarthEnv consensus land cover (1-km resolution) — 10 classes split into individual categories rather than aggregated.
  • Ocean fraction: Natural Earth 50m land polygons used to compute what fraction of each H3 cell is ocean (inverse of land area overlap).
  • Elevation: GMTED2010 mean 30-arc-second DEM — mean and standard deviation per cell.

The 12 habitat categories displayed per cell are:

11.2 Species Habitat Profiles

For each species, occurrence cells are weighted by reporting frequency to compute a habitat profile. This gives more weight to cells where the species is commonly reported, reducing the influence of stray observations.

11.3 Label Thresholds

Habitat labels are assigned when frequency-weighted averages exceed calibrated thresholds. Water-associated families (herons, ducks, grebes, loons, kingfishers, shorebirds, etc.) receive lower thresholds for Freshwater and Ocean to account for signal dilution in large hex cells.

11.4 Stixel Ensemble Habitat Similarity

The split forest types also improve the stixel ensemble's spatial smoothing. The covariate_similarity() function computes Gaussian similarity between cells using all 10 land cover dimensions plus elevation. With split forest types, a boreal conifer cell and a tropical broadleaf cell are now recognized as dissimilar habitats, suppressing inappropriate smoothing between them.

11.5 Elevation

Each species receives a preferred elevation range: the frequency-weighted mean, minimum, and maximum elevation across occurrence cells.

12. Species Photo Attribution

Species photos are sourced from Wikimedia Commons via Wikidata. The pipeline queries Wikidata for the P18 (image) property of bird taxa, then fetches metadata from the Wikimedia Commons API.

12.1 License Filtering

Only images with permissive licenses are used:

  • CC0 (Public Domain)
  • CC BY (Attribution)
  • CC BY-SA (Attribution-ShareAlike)

Images with non-commercial restrictions (CC BY-NC, CC BY-NC-SA) are excluded. Each retained photo includes full attribution: photographer name, license type, and a link to the original on Commons.

12.2 Display

Photos are hotlinked from Wikimedia Commons servers (400px thumbnail width). Species without a qualifying photo show a placeholder silhouette. Attribution text is displayed in the species information card footer.

13. Difficulty Scoring

Each species receives a difficulty score (0–100) computed from three factors derived from the eBird Basic Dataset at H3 resolution 4.

13.1 Components

Each component is rank-based: species are ranked across the full dataset, and the rank is converted to a 0–1 percentile. This ensures the full scoring range is used regardless of the underlying data distribution.

FactorWeightDescription
Spatial constraint40%Rank of cells where species occurs (fewer cells = higher difficulty rank)
Average frequency40%Rank of mean reporting frequency (lower freq = higher difficulty rank)
Seasonality20%Rank of Shannon entropy of weekly detection distribution (narrower peak = harder)

13.2 Rating Scale

The combined score is mapped to a 1–10 rating using decile thresholds, guaranteeing an even distribution (~10% of species per level). Each species also receives a text label:

14. Multi-Species Range Overlay

Range mode supports comparing 2–4 species simultaneously. Each species is assigned a distinct color from a fixed palette (blue, red, green, purple). Cells where multiple species overlap show blended colors, with gold indicating all selected species are present.

14.1 Implementation

For each selected species, per-cell reporting frequencies are loaded in parallel from the species-weeks data. A bitmask encodes which species are present in each cell (bit 0 = species 1, bit 1 = species 2, etc.). The bitmask is stored as a MapLibre feature-state value, and a match expression maps each possible bitmask to its corresponding color.

15. Gamification & Social Features

Phase 3 added celebration toasts, milestone tracking, streaks, and shareable milestone cards. Phase 4 added Firebase-backed social features.

15.1 Celebrations

Lifer celebration moments fire when species are toggled seen: success toast with count, milestone detection at thresholds (5, 10, 25, 50, 100, 250, 500, 750, 1000+), CSS confetti for medium/large milestones, and group completion trophies. Bulk CSV imports show a single summary instead of per-species toasts. Celebrations can be disabled in Profile > Preferences.

15.2 Activity Tracking

Streak tracking computes consecutive days with manual species additions (excluding bulk imports). Weekly summary shows new lifers and new family groups started. Data derived from IndexedDB dateAdded and source fields on life list entries.

15.3 Firebase Social

Email/password authentication via Firebase Auth. User profiles stored in Cloud Firestore with aggregate stats (species count, streaks, groups completed). Friend system via 6-character codes. Trip reports with species lists. Shared goal lists with real-time Firestore listeners. Leaderboards (global top 10 + friends) in Progress tab.
