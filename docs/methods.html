1. Overview & Design Philosophy

Find-A-Lifer is a progressive web app that helps birders locate species they haven't yet observed ("life birds"). It combines eBird community science data with an interactive map to show where and when species occur, at weekly temporal resolution across a multi-scale hexagonal grid.

All species occurrence data is pre-computed offline and served as static JSON files — there is no backend at runtime. This architectural choice enables offline support via service workers, instant loading from CDN/GitHub Pages, and zero server costs. The trade-off is that the data pipeline must produce complete, self-contained output files.

Why static pre-computation? The app is designed for field use by birders who may have limited connectivity. Pre-computing all data means the app works offline after initial load. It also means anyone can host a copy on GitHub Pages with no infrastructure. The pipeline processes ~50GB of raw eBird data into ~500MB of static JSON.

2. Input Data & Filtering

2.1 Data Source

All occurrence data comes from the eBird Basic Dataset (EBD), a global community science database with over 1.5 billion bird observations. Regional subsets are downloaded as paired files:

  • Sampling events file — metadata for each checklist (location, date, effort, protocol)
  • Observations file — species detections linked to sampling events

Processing is two-pass: first extract valid checklists from the sampling file, then match observations against those checklists. This avoids loading the much larger observations file into memory.

2.2 Checklist Effort Filters

Not all eBird checklists are suitable for computing reporting frequencies. We apply strict effort filters to retain only standardized, effortful surveys:

FilterCriterionValue
Date rangeObservation year1900–2025 (effectively unlimited)
ProtocolSurvey typeStationary or Traveling only
CompletenessAll species reportedYes (complete checklists only)
Duration (min)Survey length≥ 30 minutes
Duration (max)Survey length≤ 300 minutes
Distance (max)Traveling distance≤ 8 km
Why these filters?
  • Complete checklists only: Reporting frequency (detections ÷ checklists) is only meaningful when observers report all species seen, not just highlights. Incomplete lists inflate frequency for rare/exciting species and deflate it for common ones.
  • Stationary + Traveling only: These are eBird's standardized protocols with consistent effort. "Incidental" observations (seeing a bird in a parking lot) are inherently biased — the observer wasn't systematically surveying. Even with the 30-minute duration filter, we explicitly exclude incidentals to be safe.
  • 30-minute minimum: Very short checklists undercount species. A 5-minute stop detects only the most conspicuous species, artificially deflating frequencies for quieter or less visible species.
  • 300-minute maximum: Extended surveys (hawk watches, Big Sits) are valuable data, but beyond 5 hours the effort becomes qualitatively different — observer fatigue, changing conditions, etc.
  • 8 km max distance: Checklists covering long distances span too much area to be localized to a single grid cell. At our finest resolution (res 4, ~45 km hex diameter), 8 km is less than 20% of the cell width — still well-localized.
  • Stationary checklists included (distance = 0): A birder standing at a marsh or hawk watch for an hour produces excellent, perfectly-localized data. Earlier versions excluded these via a minimum distance filter, which discarded a large fraction of high-quality observations.
  • No year floor: All years are included. The Historical protocol (which would introduce life-list checklists) is already excluded by the Stationary/Traveling protocol filter. Pre-2006 data is sparse but valid.

Typically 10–30% of raw checklists pass all filters. The primary rejection reasons are incomplete lists, unsupported protocols, and duration outside bounds.

3. Spatial Grid System

3.1 H3 Hexagonal Grid

Species occurrences are aggregated onto Uber's H3 hexagonal grid at three resolution levels:

ResolutionHex DiameterMap Zoom RangePurpose
Res 2~450 km0 – 3.5Continental overview
Res 3~120 km3.5 – 5.5Regional patterns
Res 4~45 km5.5 – 22Local trip planning
Why hexagons? Hexagonal grids have uniform adjacency — every neighbor is equidistant from the center, unlike square grids where diagonal neighbors are √2× farther. This matters for spatial smoothing: when we interpolate frequencies from neighboring cells, each neighbor contributes equally in distance. H3 also provides a built-in hierarchy where each coarse cell contains exactly 7 finer cells, enabling multi-resolution fallback.
Why these resolutions? Resolution 5 (~15 km hexes) was tested but produced too many cells with sparse data, even after smoothing. Resolution 4 strikes a balance — large enough to accumulate meaningful checklist counts, small enough to distinguish local habitats. Resolution 2 provides the "zoomed out" continental view where individual cells don't need to be precise.

3.2 Ocean Handling

Unlike earlier versions that aggressively removed ocean cells, the current pipeline preserves ocean cells with direct observations — pelagic surveys and boat-based checklists produce valid bird data. Instead, ocean integrity is maintained through a multi-layer defense:

  1. Gaussian covariate similarity (§5.3): ocean↔forest similarity is ~0.006, naturally preventing land birds from bleeding into ocean cells through the ensemble.
  2. Well-sampled cell bypass (§5.2a): ocean cells with ≥10 checklists skip the ensemble entirely, using only their direct pelagic survey data.
  3. Week-limited ensemble (§5.2b): ensemble cannot fabricate data for weeks with no direct observations in a cell.
  4. Phase 2 ocean smoothing filter: neighbor smoothing does not create new data in cells with water covariate ≥ 0.9.
  5. Land-obligate species filter (§7.2): species classified as land birds are removed from ocean cells in the final output.

Environmental covariates are computed from EarthEnv 1km land cover rasters. Ocean fractions are computed separately from Natural Earth 50m land polygons by measuring the fraction of each H3 cell that does not overlap land area. Each H3 cell's ocean fraction determines whether it is treated as ocean (ocean ≥ 0.9).

3.3 Antimeridian Handling

H3 hexagonal cells near the 180° meridian (antimeridian) can produce GeoJSON polygons that wrap incorrectly — vertices jump from +179° to −179°, creating polygons that span the entire globe. The pipeline normalizes these polygons by detecting when any vertex has a negative longitude while others are positive, and shifting all negative longitudes by +360° so the polygon remains contiguous across the date line. This ensures correct rendering of cells in western Alaska and the Aleutian Islands.

4. Frequency Estimation

4.1 Reporting Frequency

The core metric is reporting frequency: the proportion of valid checklists in a cell-week that detected a given species.

frequency(species, cell, week) = detections(species, cell, week) ÷ checklists(cell, week)

This is the same metric used by eBird Status & Trends. It represents the probability that a standardized birding trip in that location and time period would detect the species. A frequency of 0.45 means 45% of qualifying checklists reported the species.

Why reporting frequency instead of abundance? The EBD contains count data (number of individuals), but abundance estimates require sophisticated detection modeling to account for imperfect detection. Reporting frequency is simpler, more robust, and directly interpretable as "chance of seeing this species on a birding trip." It naturally normalizes for observer effort — a cell with 1,000 checklists and one with 50 produce comparable frequencies.

4.2 Temporal Pooling

Many cell-week combinations have too few checklists for a reliable direct estimate. We stabilize frequencies by pooling with adjacent weeks (±1 week, wrapping at year boundaries):

For a species detected in cell C during week W:
  If checklists(C, W) ≥ 5:
    freq = detections(C, W) / checklists(C, W)        # Direct estimate
  Else:
    pooled_det = detections(C, W-1) + detections(C, W) + detections(C, W+1)
    pooled_cl  = checklists(C, W-1) + checklists(C, W) + checklists(C, W+1)
    If pooled_cl ≥ 3:
      freq = pooled_det / pooled_cl                    # Pooled estimate
    Else:
      No estimate produced (insufficient data)
Why pool adjacent weeks? Species occurrence changes gradually — a bird present in week 20 is almost certainly present in weeks 19 and 21. By borrowing checklist effort from neighboring weeks, we can produce estimates for cells that would otherwise have no data for that specific week. The 3-week window is narrow enough to preserve seasonal patterns while broad enough to fill gaps. The minimum threshold drops from 5 (direct) to 3 (pooled) because pooling itself provides some noise reduction.

4.3 Minimum Checklist Thresholds

A cell-week must have at least 5 checklists for a direct frequency estimate, or 3 checklists after temporal pooling. These thresholds prevent unreliable estimates from cells with one or two observations.

Why 5 and 3? With fewer than 5 checklists, a single observation can swing frequency from 0% to 20%+ — too noisy to be useful. The pooled threshold of 3 is lower because temporal pooling already reduces variance by combining multiple weeks. eBird S&T uses similar minimum sample sizes.

5. Stixel Ensemble Smoothing

5.1 Motivation

Even after temporal pooling, frequency estimates from individual cells can be noisy — especially in areas with moderate but not abundant eBird coverage. The stixel ensemble approach, inspired by eBird Status & Trends methodology (Fink et al. 2020), stabilizes these estimates through spatial block-averaging across multiple random partitions.

The ensemble is only applied at resolution 4. Resolutions 2 and 3 have large enough cells (~450 km and ~120 km diameter) that they naturally accumulate sufficient checklist data — averaging ~21,000 checklists per cell at res 3. Applying the ensemble at these scales would over-smooth regional patterns without meaningful noise reduction.

Why an ensemble instead of simple spatial averaging? A simple average of neighboring cells treats all neighbors equally and always smooths toward the mean. The ensemble approach creates different random spatial groupings on each run — sometimes a cell is grouped with its northern neighbors, sometimes its southern ones. Averaging across these random partitions reduces noise without systematically blurring boundaries between habitat types. Covariate weighting further ensures that ecologically similar cells contribute more to each other's estimates.

5.2 Algorithm

The ensemble runs 25 independent passes:

  1. Random partition: Shuffle all cells with data. Greedily form spatial blocks by selecting an unassigned seed cell and adding its H3 k-ring neighbors (k varies by resolution).
  2. Block-level frequency: Pool all checklists and detections within each block. Compute frequency per species per week at the block level.
  3. Weighted assignment: Assign the block frequency back to each cell in the block, weighted by environmental covariate similarity between the cell and the detection cells.
  4. Accumulate: Track the sum and count of frequency assignments per cell across all 25 runs.

After all runs, the ensemble frequency is the average across runs. This is then merged with the direct temporal-pooling estimate:

merged = 0.6 × direct + 0.4 × ensemble

Critically, if a species is present in the ensemble but not detected directly in the cell, the direct frequency is treated as 0.0 (not missing). This means ensemble-only species enter the merged estimate at only 40% of their ensemble frequency, preventing phantom species from appearing at full confidence.

5.2a Well-Sampled Cell Bypass

Cells with ≥10 total checklists (across all weeks) skip the ensemble entirely and use direct frequencies only. These cells have enough local data that ensemble smoothing would only dilute their signal. This is particularly important for preventing ocean cells with pelagic survey data from being contaminated by neighboring land cells' ensemble estimates.

5.2b Week-Limited Ensemble

The ensemble is only merged for weeks where the cell has direct observations. If a cell has checklists in summer but none in spring, the ensemble cannot fabricate spring data for that cell. This prevents the common artifact of cells showing 52-week species presence when they only have seasonal coverage.

Why 60/40 weighting? Direct observations in a cell are the most reliable signal — a bird was actually seen there. But the ensemble estimate is more stable (25 samples vs. one). The 60/40 split trusts local observations while still benefiting from the noise reduction of the ensemble.

5.3 Covariate Weighting

When assigning block-level frequencies to individual cells, each cell's contribution is weighted by how environmentally similar it is to the cells where the species was actually detected. Similarity uses a Gaussian kernel with steepness parameter k=4:

similarity = exp(−d² × 4)
where d² = Σ(cova[k] − covb[k])² over land cover types + (Δelev / 1000)²

Covariates include land cover fractions from EarthEnv (trees, shrub, herb, cultivated, urban, water, flooded vegetation) and elevation. The Gaussian kernel provides sharp discrimination between dissimilar habitats:

ComparisonSimilarityEffect
Same habitat~0.99Free sharing
Ocean ↔ coastal~0.35Shorebirds can cross
Coastal ↔ forest~0.14Moderate
Ocean ↔ forest~0.006Essentially zero
Forest ↔ plains~0.21Some generalist sharing
Why Gaussian instead of inverse distance? An earlier formula (1/(1+d×3)) compressed all similarity scores into the 0.2–0.4 range, barely distinguishing habitats. Ocean↔forest scored 0.228 — far too high, allowing land birds to bleed into ocean cells through the ensemble. The Gaussian kernel drops that to 0.006, effectively blocking cross-habitat contamination while still allowing similar habitats to share freely. If covariates are unavailable, similarity defaults to 0.5 (neutral).

5.4 Block Size

ResolutionEnsembleBlock k-ringApproximate Block Size

Res 425 runsk = 1~7 cells (~14,000 km²)
Why k=1? Larger blocks (k=2, k=3) were tested but produced over-smoothed results — rare or localized species disappeared because their signal was diluted across too many cells. A 1-ring block (~7 cells) provides enough pooling to stabilize estimates while preserving meaningful spatial variation.

6. Spatial Interpolation & Fallback

6.1 Neighbor Smoothing (Res 4 Only)

After the ensemble, resolution 4 applies one additional round of neighbor smoothing to fill cells that still have no data. For each empty cell surrounded by data cells:

For each neighbor N in 1-ring around empty cell C:
  If N has data:
    weight = spatial_weight(distance=1) × covariate_similarity(C, N)

weighted_freq(C) = Σ(freq(N) × weight(N)) / Σ(weight(N))

Smoothed cells are marked as "smoothed": 1 in the output grid and displayed at reduced opacity in the app.

Why only at res 4? Resolutions 2 and 3 have large enough cells that nearly all land area is covered by direct data. Resolution 4's smaller cells (~45 km) leave gaps, especially in less-birded areas. Neighbor smoothing fills these gaps with a conservative, distance-weighted estimate. We don't smooth at coarser resolutions because it would create data in places too far from any observations.

6.2 Multi-Resolution Independence

Each resolution stands on its own data — there is no downward fallback between resolutions. Resolution 2 is aggregated upward from resolution 3 archive data, but resolution 3 does not fall back to resolution 2, and resolution 4 does not fall back to resolution 3.

Why no fallback? Earlier versions used parent-resolution data (with a 30% discount) to fill empty child cells. This caused land bird species to propagate into ocean cells — a res 3 coastal cell containing both land and ocean would push its full species list down into all seven res 4 children, including ocean-only cells. With the ensemble already filling gaps at res 4, fallback was both unnecessary and harmful.

7. Thresholds & Quality Control

7.1 Minimum Smoothed Frequency

Species-cell entries from interpolated (non-direct) cells with frequency below 0.5% are dropped:

If cell is interpolated AND freq < 0.005: discard

This threshold is not applied to direct-detection cells — if a species was actually observed in a cell, it stays regardless of computed frequency.

Why exempt direct detections? A rare vagrant observed once in 200 checklists has a 0.5% frequency — low, but real. Dropping it would erase genuine records. For interpolated cells, though, a 0.3% frequency likely represents noise propagated through the ensemble and smoothing. The threshold strikes a balance between coverage and credibility.

7.2 Land-Obligate Species Filter

To prevent land birds (e.g., Ravens, woodpeckers) from appearing in ocean cells through ensemble bleeding or sparse pelagic checklist noise, a land-obligate species filter is applied in the output phase:

  1. For each species, compute the average water covariate across all cells where it occurs.
  2. Species belonging to waterbird families (gulls, ducks, loons, alcids, shorebirds, cormorants, pelicans, storm-petrels, shearwaters, boobies, frigatebirds, tropicbirds, skuas, oystercatchers, grebes) are exempt — always allowed in ocean cells.
  3. Non-exempt species with average water < 0.3 are classified as land-obligate.
  4. Land-obligate species are removed from cells with water ≥ 0.9.
Why family exemption? Many waterbirds (Herring Gull, Common Loon, scoters) are commonly reported from harbors and lakeshores — cells with low water covariates. Their average water score drops below 0.3 despite being perfectly legitimate ocean species. The family-level exemption ensures all waterbird families pass through regardless of their covariate statistics. For non-waterbird species, the avg_water < 0.3 threshold catches true land birds like passerines, raptors, and owls.

7.3 Species Minimum Record Threshold

Species with fewer than 10 total records across all data are excluded from output. This prevents noisy frequency estimates from very rare vagrants or data-entry errors that lack sufficient observations for meaningful spatial or temporal patterns.

A force-keep list preserves genuine island endemics and range-restricted species regardless of record count. These include species like Clarion Wren, Cozumel Thrasher, and Guadalupe Junco — genuinely rare endemics whose low observation counts reflect true rarity within the coverage area rather than data quality issues.

This threshold dropped 146 species from the output. These species are saved separately for future reintegration as the app's geographic coverage area expands and more eBird data becomes available for currently under-sampled regions.

Why 10 records? A species observed only 3–5 times across all cells and weeks produces extremely noisy frequency estimates that mislead trip planners. The threshold of 10 records provides a minimal statistical basis for spatial patterns while remaining low enough to retain legitimately uncommon species. The force-keep list ensures that genuine endemics with inherently low observation counts are never lost.

7.4 Escapee Species Filter

The eBird Basic Dataset includes an EXOTIC CODE column that classifies non-native observations. The pipeline now distinguishes between the two provisional/escapee codes:

  • P (Provisional) — species with established or potentially establishing populations. These are real vagrants and naturalized populations (e.g., Eurasian Collared-Dove expanding its range). Kept in the dataset.
  • X (Escapee) — species that are almost certainly escaped captive birds, zoo escapes, or released pets (e.g., a parrot species far outside its native range with no breeding population). Filtered out when escapee in all regions.

Species that carry the X (Escapee) exotic code in every region where they occur are removed from the final output. This eliminates 140 species that represent escaped pets, zoo birds, and other non-wild observations, reducing the species count from 2,307 to 2,167. Species that are native or provisional in at least one region are retained with their full occurrence data.

Why filter escapees globally rather than per-region? A species that is genuinely wild somewhere in the coverage area (native or provisional in any region) should remain available for all regions — it may be an escapee in one region but a legitimate vagrant target in another. Only species that are exclusively escapees everywhere are removed. This conservative approach avoids accidentally dropping species like Monk Parakeet (escapee origin but with established breeding colonies) while still removing species like Indian Peafowl that have no wild populations anywhere in the coverage area.

8. Output Format

8.1 Frequency Encoding

Frequencies (0.0–1.0) are encoded as unsigned 8-bit integers (0–255) for compact storage:

freq_uint8 = max(1, min(255, round(freq × 255)))

Client-side decoding: probability = freq_uint8 / 255. A value of 127 represents ~50% frequency.

Why uint8? The raw frequency data for all species, cells, and weeks would be hundreds of megabytes as floating-point JSON. Quantizing to 0–255 roughly halves file sizes while preserving more than enough precision for visualization and trip planning. The ~0.4% granularity (1/255) is well below the noise floor of the underlying estimates.

8.2 File Structure

All output is organized per resolution in data/r{2,3,4}/:

FileContentsPurpose
grid.geojsonH3 hex polygons with cell_id, center coords, label, smoothed flagMap overlay geometry
weeks/week_XX_summary.json[cell_id, species_count, max_freq, n_checklists] per cellFast heatmap rendering
weeks/week_XX_cells.json[cell_id, [species_ids...], [freq_uint8...]] per cellCell click → species list
species-weeks/{code}.json{week: [[cell_id, freq], ...]}Single-species range maps

A shared species.json contains metadata for all 2,167 species across 13 countries and 31 eBird regions (AG, AW, BB, BM, BQ, BS, BZ, CA, CR, CU, DM, DO, GD, HT, JM, KN, LC, MF, MQ, MX, NI, PA, PM, PR, SX, TC, TT, US, VC, VG, VI) on a grid of 10,607 cells at resolution 4. Fields include: common name, scientific name, family, taxonomic order, eBird species code, conservation status, invasion status per region, difficulty rating, regional difficulty, habitat labels, and photo attribution.

Invasion status is derived from the EBD EXOTIC CODE column: empty → Native, N → Introduced, P → Provisional (real vagrants, kept), X → Escapee (filtered out when escapee in all regions; see §7.4). Stored per-species as Record<string, string> keyed by region code. A native override rule applies: if ANY native checklist exists for a species in a region, the species is classified as Native in that region regardless of any exotic-coded checklists. This prevents misclassification of native species that occasionally appear on exotic lists.

Display priority for invasion badges: Native > Introduced > Vagrant/Accidental. If a species is native in any region within the current scope (global or filtered region), no invasion badge is shown — native status takes precedence. This respects the historical timeframe the bird has been present. When a region filter is active, only invasion entries for that region's country codes are considered.

Why separate summary and cell files? The summary file (~2 MB/week) contains just enough to render the heatmap — species count and max frequency per cell. The full cell file (~30-50 MB/week) contains the complete species list with individual frequencies, but is only loaded on demand when the user clicks a cell. This lazy-loading pattern keeps the initial page load fast while still providing full detail when needed.

9. Client-Side Computation

9.1 Lifer Count (formerly Richness / Lifer Density)

The "Count" view mode subtracts the user's life list from each cell's species count:

lifer_count(cell, week) = total_species(cell, week) − seen_species(cell, week)

9.2 Combined Lifer Probability

For the "Chance" view (formerly Frequency), the app computes the probability of observing at least one unseen species during a visit:

P(≥1 lifer) = 1 − ∏i(1 − freqi)   for all unseen species i in the cell
Why this formula? Under the assumption that detecting each species is approximately independent (reasonable for well-separated taxa), the probability of detecting none of them is the product of the individual non-detection probabilities. One minus that product gives the probability of at least one detection. This is the most useful metric for a lifer-seeking birder: "if I go here, what's my chance of adding something new to my list?"

9.3 Resolution Switching

The app automatically switches between resolutions as the user zooms:

  • Zoom 0–3.5: Resolution 2 (~450 km hexes)
  • Zoom 3.5–5.5: Resolution 3 (~120 km hexes)
  • Zoom 5.5+: Resolution 4 (~45 km hexes)

Each resolution has independently computed data, so switching is seamless — the appropriate grid and weekly data are loaded fresh for the new resolution.

9.4 Quantile Normalization

All five view modes (Count, Chance, Range, Goals, multi-species Range) use quantile normalization for the heatmap color gradient, replacing the earlier linear value/max scaling. Instead of mapping raw values linearly to the color ramp, cells are ranked by percentile and spread evenly across the gradient:

color_position = rank(cell_value) / total_cells_with_data

This ensures the full color range is always utilized regardless of the underlying value distribution. With linear scaling, a few extreme outlier cells could compress 95% of cells into the bottom of the color range, making most of the map appear uniformly low. Quantile normalization guarantees an even visual spread: the top 10% of cells always appear in the top 10% of the color gradient.

Raw values are preserved and displayed in cell popups and tooltips, so users still see the actual species count, probability, or frequency — only the map coloring uses percentile ranks.

Why quantile instead of linear? Species count and frequency distributions are heavily right-skewed: most cells have modest values while a few hotspot cells have very high counts. Under linear scaling, the hotspots dominate the color range and everything else looks the same shade of pale. Quantile normalization reveals spatial patterns across the full range of values, making it much easier to distinguish "good" from "great" birding locations.

9.5 Goal Birds Combined Probability

The Goal Birds view mode now displays combined probability rather than a simple species count. For cells containing goal list species, the probability is computed identically to the Chance view:

P(≥1 goal bird) = 1 − ∏i(1 − freqi)   for all goal species i in the cell

This gives a more actionable metric than a raw count — a cell with 3 goal species at 5% frequency each is less productive than a cell with 1 goal species at 60% frequency. The probability view reflects the actual likelihood of encountering at least one goal bird during a visit.

10. Grid Cell Labeling

Each grid cell is labeled with the nearest recognizable city for user orientation. The labeling algorithm balances proximity, city size, and label diversity:

10.1 Scoring

score = population / (distance_km + 1)1.5 × region_boost

Where region_boost = 1.5 if the city and cell are in the same state/province, else 1.0. The search radius is resolution-dependent (120 km for res 3, 60 km for res 4).

10.2 Deduplication

To prevent a single major city (e.g., New York City, Boston) from labeling dozens of surrounding cells, each city is limited to a maximum number of cells per resolution (2 at res 3, 3 at res 4). City name variants ("South Boston," "East Boston," "North Boston") are grouped under the base name for deduplication.

Why population-weighted scoring? Users need recognizable landmarks for orientation. "Near Boston" is more useful than "near Framingham" even if Framingham is slightly closer. The 1.5 exponent on distance ensures proximity still matters — a distant megacity won't dominate a cell right next to a midsized town. The same-state boost prevents border cells from being labeled with out-of-state cities (e.g., a New Hampshire cell getting a "Boston, MA" label).

11. Habitat Classification

Each species is assigned habitat labels based on its occurrence patterns across environmental covariates. The pipeline (compute_species_habitat.py) computes frequency-weighted average covariates for each species across all cells and weeks where it occurs.

11.1 Covariate Sources

Cell-level environmental covariates are extracted from two sources:

  • Land cover: EarthEnv consensus land cover (1-km resolution) — 10 classes split into individual categories rather than aggregated.
  • Ocean fraction: Natural Earth 50m land polygons used to compute what fraction of each H3 cell is ocean (inverse of land area overlap).
  • Elevation: GMTED2010 mean 30-arc-second DEM — mean and standard deviation per cell.

The 12 habitat categories displayed per cell are:

11.2 Species Habitat Profiles

For each species, occurrence cells are weighted by reporting frequency to compute a habitat profile. This gives more weight to cells where the species is commonly reported, reducing the influence of stray observations.

11.3 Label Thresholds

Habitat labels are assigned when frequency-weighted averages exceed calibrated thresholds. Water-associated families (herons, ducks, grebes, loons, kingfishers, shorebirds, etc.) receive lower thresholds for Freshwater and Ocean to account for signal dilution in large hex cells.

11.4 Family Display Groups

Species are organized into display groups that map eBird's 115 taxonomic families into user-friendly categories. After the Caribbean expansion (adding 15 island territories), 32 new families were mapped into display groups. Notable group changes include:

  • Tyrant Flycatchers renamed to Flycatchers for brevity
  • Swifts and Swallows merged into a single display group
  • Cuckoos expanded to Cuckoos and Allies to include related families
  • Sandgrouse folded into Pigeons display group

The total display group count remains stable at ~40 groups, with ecological ordering preserved (waterfowl first through passerines).

11.5 Stixel Ensemble Habitat Similarity

The split forest types also improve the stixel ensemble's spatial smoothing. The covariate_similarity() function computes Gaussian similarity between cells using all 10 land cover dimensions plus elevation. With split forest types, a boreal conifer cell and a tropical broadleaf cell are now recognized as dissimilar habitats, suppressing inappropriate smoothing between them.

11.6 Elevation

Each species receives a preferred elevation range: the frequency-weighted mean, minimum, and maximum elevation across occurrence cells.

12. Species Photo Attribution

Species photos are sourced from Wikimedia Commons via Wikidata. The pipeline queries Wikidata for the P18 (image) property of bird taxa, then fetches metadata from the Wikimedia Commons API.

13. Difficulty Scoring

Each species receives a difficulty score (0–100) computed from three factors derived from the eBird Basic Dataset at H3 resolution 4.

13.1 Components

Each component is rank-based: species are ranked across the full dataset, and the rank is converted to a 0–1 percentile. This ensures the full scoring range is used regardless of the underlying data distribution.

FactorWeightDescription
Spatial constraint40%Rank of cells where species occurs (fewer cells = higher difficulty rank)
Average frequency40%Rank of mean reporting frequency (lower freq = higher difficulty rank)
Seasonality20%Rank of Shannon entropy of weekly detection distribution (narrower peak = harder)

13.2 Rating Scale

The combined score is mapped to a 1–10 rating using decile thresholds, guaranteeing an even distribution (~10% of species per level). Each species also receives a text label:

RatingLabelExample Species
1–2EasyAmerican Robin (1), Mallard (1), Bald Eagle (1)
3–4ModerateRed Crossbill (3), Horned Grebe (3)
5–6HardBobolink (5), Resplendent Quetzal (6)
7–8Very HardIvory Gull (7)
9–10Extremely HardRarest vagrants and extreme specialists

13.3 Per-Sub-Region Difficulty

The same rank-based percentile methodology is applied independently within each of the 17 sub-regions (see §15) and also within 5 super-regions (Northern, Continental US, Hawaii, Mexico & Central America, Caribbean). Super-regions provide a broader geographic context that captures species rarity at the biome level rather than the political boundary level. A species that is common in one region may be extremely rare in another, producing dramatically different scores. Regional difficulty is stored in species.regionalDifficulty as Record<subRegionId, number>, and super-region difficulty in species.superRegionDifficulty as Record<superRegionId, {difficultyScore, difficultyRating, difficultyLabel}>. Cell assignment to sub-regions uses state codes from cell_states.json.

13.4 Manual Overrides

Both the conservation status and difficulty pipelines support manual override files that are applied as the final step, taking precedence over all computed values:

  • Conservation overrides (pipeline/reference/conservation_overrides.json): 161 species with manually corrected IUCN status codes, applied after the Wikidata SPARQL → IUCN → NatureServe → COSEWIC priority merge. Corrects species where Wikidata is outdated due to taxonomic splits or where the automated merge produces incorrect results (145 LC, 5 NT, 4 VU, 6 EN, 1 CR).
  • Difficulty overrides (pipeline/reference/difficulty_overrides.json): 88 species with manually set difficulty scores and ratings, overriding the rank-based computation. Corrects for pelagic species (inherently hard but data-sparse), nocturnal/skulking species, Hawaiian endemics, range-restricted endemics, introduced species (artificially inflated difficulty from small ranges), and data artifacts.

14. Multi-Species Range Overlay

Range mode supports comparing 2–4 species simultaneously. Each species is assigned a distinct color from a fixed palette (blue, red, green, purple). Cells where multiple species overlap show blended colors, with gold indicating all selected species are present.

14.1 Implementation

For each selected species, per-cell reporting frequencies are loaded in parallel from the species-weeks data. A bitmask encodes which species are present in each cell (bit 0 = species 1, bit 1 = species 2, etc.). The bitmask is stored as a MapLibre feature-state value, and a match expression maps each possible bitmask to its corresponding color.

15. Sub-Regions & Regional Species Cards

Species cards are region-aware. When opened from a map click, the card auto-detects the geographic sub-region and filters all displayed data (seasonality, difficulty, best locations, invasion status) to that region.

15.1 Sub-Region Definitions

17 sub-regions are defined by state/province codes (not bounding boxes). Cell assignment uses STATE CODE data captured from the EBD during pipeline processing, stored as cell_states_r{N}.json and shipped to the frontend as r{N}/cell_states.json. Each cell maps to a state code, which in turn maps to a sub-region via STATE_TO_REGION lookup.

Why state codes instead of bounding boxes? Bounding boxes cannot accurately represent irregular political boundaries — Alaska's Aleutian Islands extend west of the date line, Florida wraps around the Gulf coast, and Caribbean island nations are scattered across the sea. State codes from the EBD provide exact assignment because every eBird checklist carries a STATE CODE field. The pipeline captures this per cell, enabling precise sub-region membership. For Caribbean countries where province codes include sub-codes (e.g., CU-03), prefix matching extracts the country code (CU).

15.2 Regional Difficulty

Difficulty ratings are pre-computed per sub-region using the same rank-based percentile approach as global scores (decile 1–10), but scoped to cells within each region. This produces dramatically different ratings: a Rusty Blackbird scores 3/10 in Eastern Canada but 10/10 in Mexico. Stored in species.regionalDifficulty as a Record<subRegionId, number>.

15.3 Regional Habitat Labels

Habitat labels are computed at three granularity levels: global, per-sub-region, and per-super-region. A species may be labeled "Forest" globally but "Wetland" in a specific region where it predominantly uses different habitat. All levels use the same frequency-weighted covariate methodology, scoped to the appropriate cell set.

Super-region habitat is computed across 5 broad geographic zones (Northern, Continental US, Hawaii, Mexico & Central America, Caribbean) and stored in species.superRegionHabitat. This provides meaningful habitat context even for species with sparse sub-region data.

15.4 Regional Data Filtering

When a sub-region is selected, the species card filters:

  • Seasonality sparkline — only cells within the region (via state code matching)
  • Best locations — filtered to region (with "Show global" toggle)
  • Difficulty — regional rating from pre-computed data
  • Habitat labels — regional habitat profile
  • Invasion status — only shows status for the region's country codes

A dropdown on the card shows all regions where the species has occurrence data, allowing manual switching. The dropdown is populated by scanning the species-weeks data for cells in each sub-region via cell_states mapping.

15.5 Region Filter Integration

Sub-regions are fully integrated into the species filter dropdown across all tabs. The region filter system (regionGroups.ts) defines 17 sub-regions plus multi-country groups:

  • US sub-regions (7 + Hawaii): Northeastern, Southeastern, Midwestern, Southwestern, Western US, US Rockies, Hawaii — shown under "United States" optgroup
  • Canada sub-regions (3): Western Canada & Alaska, Central Canada, Eastern Canada & North Atlantic Islands — shown under "Canada" optgroup
  • Mexico sub-regions (2): Northern, Southern Mexico — shown under "Mexico" optgroup
  • Central America (2): Upper Central America (BZ/GT/SV/HN/NI), Southern Central America (CR/PA) — shown under "Central America" optgroup
  • Caribbean (3): Greater Antilles, Lesser Antilles (15 territories), Western Atlantic Islands — shown under "Caribbean" optgroup

Two-tier filtering: selecting a sub-region like "Northeastern US" expands to the country code ['US'] for species-level filtering (matching species.json regions), while the sub-region's bounding box provides cell-level masking on the map. This means the species list shows all US species but the heatmap only highlights cells within the Northeastern US bounding box.

The Progress tab uses species.regionalDifficulty keys as sub-region presence indicators to compute per-sub-region species counts, avoiding the ambiguity of mapping a single country code (US) to multiple sub-regions.

16. Goal Birds Recommendation Engine

An app-wide intelligent recommendation system helps users discover species to target. The engine (recommendationEngine.ts) analyzes the user's life list, current week, and species metadata to surface contextual suggestions across multiple parts of the UI.

16.1 Notable Birds (Map Popups)

When a user clicks a cell, the popup highlights notable unseen species in three priority tiers:

  1. Goal Birds — species on the user's active goal list that occur in this cell
  2. Rare Finds — species with difficulty ≥ 7 that the user hasn't seen
  3. Easy Wins — species with high reporting frequency (≥ 30%) and difficulty ≤ 3

A minimum frequency filter of 1% (freq < 0.01) is applied to all notable birds. Species with reporting frequencies below this threshold are excluded from notable recommendations to avoid surfacing extremely unlikely sightings that would mislead trip planning.

An "all species" popup mode shows the complete cell species list with checkmarks for already-seen species. Users can add species to their goal list directly from Notable Birds entries and species cards.

16.2 This Week's Highlights

Time-sensitive recommendations surface in two places: the Explore tab (desktop) and the floating MapControls overlay (mobile). Four categories of highlights are computed:

  • Best Chance (Goal) — goal list species near their annual peak week
  • New Arrivals — species arriving soon (peak 1–3 weeks away) with moderate-to-high difficulty
  • Peak Season — unseen species at their annual frequency maximum this week
  • Rare Visitors — difficult species (rating ≥ 7) at or near peak

On desktop, the full-frequency version (getWeeklyHighlights()) uses week cell data for precise frequency comparisons. On mobile, a lightweight version (getWeeklyHighlightsLite()) uses only the peakWeek field from species metadata, avoiding expensive data fetches while still surfacing actionable highlights. The mobile highlights card appears as a collapsible "This Week" section below the week slider in MapControls.

16.3 Smart Goal Suggestions

Auto-generated goal suggestions based on life list analysis. The engine selects the 3–4 most relevant suggestion sections from:

  • Family completion — families where the user has seen most but not all species
  • Easy lifers — common, widespread species the user hasn't yet seen
  • Seasonal opportunities — species with narrow seasonal windows currently in peak
  • Colorful characters — curated list of 29 visually striking species
  • Regional icons — curated specialists from 10 regions (including Florida, Texas, Alaska, Caribbean)

16.4 Goal Templates

Pre-built goal lists for targeted birding challenges:

CategoryTemplates
ConservationThreatened species (VU/EN/CR), Near Threatened, Data Deficient, Invasive species, Restricted-range (≤ 2 regions)
DifficultyEasy lifers (rating 1–2), Hardest birds (rating 9–10)
HabitatForest specialists, Ocean/pelagic, Wetland specialists

Why restricted-range ≤ 2 regions? Species occurring in only 1–2 of the app's 31 regions are genuinely range-restricted within the coverage area. This is computed from the invasionStatus keys (which track which regions a species occurs in as native), giving a proxy for range size without requiring additional data.

16.5 Accessibility

Goal bird UI components use ARIA attributes throughout: aria-label on interactive elements, aria-expanded on collapsible sections, and role="progressbar" on completion indicators.

17. Trip Planning

The Trip Plan tab provides four modes for planning birding trips using the pre-computed species occurrence data. All modes operate on the user's life list and selected week to surface the most productive locations and timing. When a region filter is active, all four modes restrict their species lists to the selected region's country codes, ensuring results are geographically relevant.

17.1 Hotspots

Ranks grid cells by lifer count for the selected week. Uses computeLiferSummary() to count unseen species per cell, producing a ranked list of the most productive locations for finding new life birds. Results show cell name, lifer count, and top unseen species. Clicking a result centers the map on that cell.

17.2 Window of Opportunity

Two sub-modes for finding optimal timing:

  • Single species — given one species, finds the peak frequency weeks and best locations across the year. Uses species-weeks frequency profiles to identify when and where the species is most detectable.
  • Goal list — given the user's goal list, finds week + location combinations that maximize goal species coverage. Scans all weeks and cells to surface trips where multiple goal species overlap in time and space.

17.3 Compare Locations

Side-by-side comparison of two grid cells for the selected week. Shows three sections:

  • Overlap species — species present in both cells, with frequency data from each
  • Unique to A — species found only in the first cell
  • Unique to B — species found only in the second cell

Helps users decide between two potential trip destinations by comparing what each offers.

17.4 Location Analysis

Detailed species breakdown for a single cell during a date range. Shows all species expected in the cell across the selected weeks with reporting frequencies, grouped by family. Useful for building a target list before visiting a specific location.

19. Pipeline Orchestrator

The full data pipeline involves seven sequential steps that must run in order, with each step depending on the outputs of the previous one. To eliminate manual orchestration errors, a dedicated orchestrator script (pipeline/run_all.py) chains all steps with validation gates between each stage.

19.1 Step Sequence

process_ebd.py
→
label_cells.py
→
extract_covariates.py
→
compute_ocean_fraction.py
→
ship_covariates.py
→
compute_difficulty.py
→
compute_species_habitat.py

After each step completes, a validation gate checks that expected output files exist and are non-empty. For example, after process_ebd.py, the gate verifies that species.json, all grid.geojson files, week data directories, and archive files are present. After label_cells.py, it spot-checks that grid features contain the label property. If any validation fails, the pipeline halts immediately with a clear error message rather than allowing downstream steps to operate on missing or stale data.

19.2 CLI Options

FlagEffect
--rebuildPasses --rebuild to process_ebd.py to regenerate output from archived intermediate counts (no new EBD files needed)
--skip-ebdSkips step 1 entirely — useful when EBD processing is already done and only post-processing steps need re-running
--no-stixelPasses --no-stixel to process_ebd.py to skip the ensemble smoothing pass (faster iteration during development)
--dry-runPrints the ordered step list with commands without executing anything — useful for verifying what will run before committing

Each step is timed individually, and the orchestrator reports a summary with per-step durations and pass/fail status at the end of a run.

Why an orchestrator? Before run_all.py, the seven pipeline steps had to be run manually in order. Skipping or misordering a step (e.g., running compute_difficulty.py before ship_covariates.py) produced subtle data inconsistencies that were difficult to diagnose. The orchestrator with validation gates makes the pipeline a single command with fail-fast behavior.

19.3 Species-Habitat Affinity Spike

A feasibility spike tested whether per-species habitat profiles could improve gap-filling over the current cell-to-cell covariate similarity approach. The hypothesis: if a species prefers deciduous forest, cells with deciduous forest should receive higher smoothing weight regardless of proximity to detection cells.

For each species, a weighted-average habitat profile was computed from all detection cells (weighted by detection count). Non-detection cells were then scored by two methods:

  1. Cell-to-cell similarity (current): Gaussian similarity to the nearest detection cell's covariates
  2. Cell-to-profile similarity (proposed): Gaussian similarity to the species' habitat profile

Results across 19 species (2,400–6,800 detection cells each): the profile-based approach added 0% improvement over cell-to-cell similarity, which won 78% of comparisons on average. For widespread species, every non-detection cell already has a nearby detection cell with similar habitat, making the averaged profile too generic to add value.

Conclusion: The current cell-to-cell covariate similarity is sufficient for the ensemble smoothing. The real data quality improvements will come from the stixel rebuild (fixing ocean cell bleeding, currently affecting 1,121 species at res 4) and quantile normalization (fixing the color scale). Full encounter rate modeling with random forests (using covariates as predictive features rather than similarity weights) remains a viable Sprint 5+ research direction.

20. Frontend Performance

Several performance and resilience patterns keep the app responsive on mobile devices and prevent unbounded resource consumption.

20.1 Error Boundaries

Three React error boundaries isolate failures to their respective domains:

  1. App root boundary — catches top-level rendering failures and displays a recovery UI
  2. Map boundary — isolates MapLibre GL crashes (WebGL context loss, tile errors) so the side panel remains functional
  3. Tab boundary — wraps individual tab content so a failure in one tab (e.g., Goals) does not take down the entire side panel

20.2 Code Splitting

Heavier tab components are loaded on demand via React.lazy() with Suspense fallbacks. The lazily loaded tabs are:

  • Goal Birds tab
  • Trip Plan tab
  • Stats tab

The Explore and Species tabs are included in the main bundle since they are the default entry points. The app uses 5 tabs (Explore, Species, Goals, Plan, Stats) — Profile settings have been consolidated into the kebab menu. This reduces the initial JavaScript payload and speeds up first paint on slower connections.

20.3 LRU Cache Eviction

The data cache (dataCache.ts) uses an LRU (Least Recently Used) eviction strategy to prevent unbounded memory growth as users browse across weeks and resolutions. Week summary and cell data are evicted when the cache exceeds a configured entry limit, keeping the most recently accessed data in memory while freeing older entries. This is particularly important on mobile devices with constrained memory budgets.

20.4 No-Data Cell Feedback

When a user clicks a cell that has no species data for the selected week, a toast notification explains the situation rather than showing an empty or confusing popup. This provides clear feedback instead of silent failure.

20.5 Touch Targets

All mobile map controls (zoom buttons, view mode selector, week slider) use a minimum 44×44px touch target, following WCAG 2.5.5 (Enhanced) guidelines. This ensures usability on touchscreens without requiring precise taps.

20.6 Resizable Desktop Sidebar

On desktop viewports, the side panel features a draggable left edge for resizing between 320px and 800px width. The resize handle appears as a subtle grip indicator on hover. The user's preferred width is persisted in localStorage and restored on subsequent visits. This allows users to expand the panel for detailed species lists or narrow it to maximize the map area.

20.7 MapControlsContext

Map control state (view mode, week, opacity, filters) was previously passed through 5+ levels of prop drilling from App.tsx down through SidePanel, ExploreTab, and MapControls into MapView. A dedicated React context (MapControlsContext.tsx) now provides this state directly to any component that needs it, eliminating prop drilling and simplifying component interfaces across 12+ files.

20.8 Guided Feature Tour

New users are guided through the app with a driver.js feature tour (lib/featureTour.ts) that replaces the previous onboarding overlay slides. The tour consists of 8 steps highlighting key UI elements:

  1. Map overview (hexagonal cells)
  2. View mode selector (Count/Chance/Range/Goals)
  3. Week slider for temporal exploration
  4. Cell click interaction
  5. Species tab for browsing and life list management
  6. Goals tab for goal list creation
  7. Plan tab for trip planning
  8. Menu button (import, dark mode, tutorial, about)

The tour is responsive: on mobile (<768px), steps target the bottom tab bar buttons and floating MapControls; on desktop, they target the side panel tab navigation. driver.js is lazy-loaded via dynamic import() to avoid increasing the main bundle size. Tour completion state is persisted in localStorage and can be replayed from the kebab menu.

20.9 Geolocation Prompt

After a user's first life list import, the app prompts for browser geolocation permission. If granted, the map flies to the user's location at zoom level 7 (resolution 4, local trip planning scale). This only triggers when no saved map position exists in localStorage, ensuring returning users are not re-prompted.

20.10 Font Size Accessibility

A minimum font size of 12px is enforced throughout the app. A responsive three-tier type scale ensures readability across devices: primary text uses 12px on mobile scaling to 14px on large screens, while icon-associated text uses 11px scaling to 12px. No text in the app is rendered below 11px.

20.11 Manual Species Add

Users can add individual species to their life list via a search-and-add modal accessible from the kebab menu. The modal provides a fuzzy search across all species names (common and scientific), allowing users to quickly add species they have seen without importing a full CSV file. Each addition triggers the same celebration and milestone logic as other life list additions.

21. PWA & Installability

Find-A-Lifer is a progressive web app installable on Android, iOS, and desktop. The service worker (via vite-plugin-pwa with Workbox) pre-caches the app shell and lazily caches data files for offline use.

App icons are generated as PNG files at multiple sizes (192×192, 512×512) for the web manifest, ensuring compatibility with Android home screen shortcuts, iOS "Add to Home Screen," and desktop PWA install prompts. Maskable icon variants are included for Android adaptive icon rendering.

23. Firestore Security Rules

Cloud Firestore security rules enforce authentication-based access control for all social features. Each user's profile, friend connections, trip reports, and shared goal lists are scoped to their Firebase Auth UID. Rules ensure:

  • Users can only read and write their own profile data
  • Friend connections require mutual consent (both users must accept)
  • Shared goal lists are readable by participants but only editable by the owner
  • Leaderboard entries are read-only for non-owners, preventing score manipulation
